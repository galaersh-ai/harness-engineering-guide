# Команды агентов: параллельные Claude строят реальное ПО

Большинство демо агентов выглядят впечатляюще и ничего полезного не делают. Приложение-список задач. Скрейпер todo. «Личный исследовательский ассистент», суммирующий три веб-страницы.

Затем кто-то подключает шестнадцать Claude в цикл, и они строят C-компилятор, способный компилировать Linux-ядро.

Эта статья о втором — и specifically, о том, как выглядел харнес под ней. Мы пройдём по архитектуре из проекта Nicholas Carlini *«Building a C compiler with a team of parallel Claudes»* ([Anthropic Engineering](https://www.anthropic.com/engineering/building-c-compiler), [GitHub](https://github.com/anthropics/claudes-c-compiler)), выделим принципы проектирования, которые действительно mattered, и переведём их в паттерны, которые Вы можете переиспользовать в собственном харнесе.

---

## Ключевая идея

> **16 экземпляров Claude, работающих параллельно около двух недель, создали C-компилятор на 100 000 строк Rust. Он компилирует Linux 6.9 на x86, ARM и RISC-V. Он проходит 99% тестового набора GCC torture. Он компилирует QEMU, FFmpeg, SQLite, Postgres и Redis.**

Прочитайте это ещё раз. Затем ещё раз.

Это не игрушка. Это реальный production-компилятор, написанный командой LLM под ограничениями, которые человеческая команда не выдержала бы: без доступа в интернет, без существующих компиляторных crate, без копирования из LLVM или tcc — только стандартная библиотека Rust и всё, что Claude мог вывести из C-спецификации и своего обучения. Чистая реализация, сделанная агентами.

Интересный вопрос не «может ли LLM написать компилятор». Мы уже знали, что может написать *части* интересный вопрос:

**Какой харнес превращает 16 бесостоятельных, забывчивых вызовов LLM в согласованную инженерную команду, выпускающую 100 000 строк кода?**

## Архитектура Ralph-Loop

Сердце харнеса позорно простое. Каждый агент — это bash-цикл:

```bash
#!/usr/bin/env bash
# agent.sh — run inside a Docker container
while true; do
  claude --dangerously-skip-permissions -p AGENT_PROMPT.md
done
```

Вот и всё. Вот весь агент.

Нет оркестратора. Нет демона очереди задач. Нет планировщика. Нет шины сообщений. Когда одна сессия Claude завершается — успех, падение или collapse контекстного окна — цикл запускает свежий Claude с тем же файлом промпта. Промпт говорит новому Claude пойти посмотреть в репозиторий, выяснить, что нужно сделать дальше, и сделать это.

Этот стиль иногда называют «Ralph loop» (в честь Ralph Wiggum и его «Я помогаю!»): глупый внешний цикл продолжает пинать умный внутренний процесс, пока работа не будет сделана.

Единственное вмешательство уровня супервайзера:

```bash
# kill the fleet
pkill -9 bash
```

Это весь протокол завершения. Нет мягкого дрейфа, нет «пожалуйста, завершите текущую задачу». Вы SIGKILL'ите bash-циклы, и агенты умирают на полпути мысли. В следующий раз, когда Вы запустите их, они продолжат с того места, где `git pull` их оставил.

### Почему это работает

Ralph loop — правильная форма для работы агента по трём причинам:

1. **LLM и так бесостоятельны между сессиями.** Притворяться, что они — долгоживущие процессы, — значит просто прятать collapse контекстного окна за всё более отчаянными трюками.
2. **Перезапуск бесплатен.** Свежий Claude, читающий репозиторий, дешевле застойного Claude, заполнившего свой контекст 3 часами неудачных экспериментов.
3. **Изоляция сбоев приходит бесплатно.** Если сессия уходит в створку, следующая сессия не наследует обломки — она наследует git-состояние, которое является единственным источником правды.

## Координация через git (без оркестратора)

Вот что удивляет большинство людей: **нет центрального оркестратора.** Ничто не раздаёт задачи. Ничто не знает, кто что делает. Нет дашборда с «агент-7 сейчас работает над struct-packing».

Вместо этого флот координируется через два примитива:

1. **Голый git-репозиторий**, в который каждый контейнер пушит и из которого пуллит.
2. Директория `current_tasks/`, содержащая **lock-файлы** — по одному на задачу в процессе, записанные как `current_tasks/<task-name>.txt`.

Каждый промпт агента по сути говорит:

> «Посмотри на открытые TODO, падающие тесты и файлы прогресса. Выбери следующую самую очевидную задачу, над которой сейчас никто не работает. Оставь lock-файл. Сделай работу. Закоммить. Удали lock. Выйди.»

Вот жизненный цикл за сессию:

```
 ┌──────────────────────────────────────────────────────────────┐
 │  1. git clone / pull                                         │
 │  2. scan repo → pick task T                                  │
 │  3. git add current_tasks/T.txt   ← claim the lock           │
 │  4. git push                      ← publishes the claim      │
 │  5. work on T (write code, run tests, commit)                │
 │  6. git pull --rebase             ← grab peers' changes      │
 │  7. resolve merges / re-run tests                            │
 │  8. git push                      ← ship it                  │
 │  9. git rm current_tasks/T.txt    ← release lock             │
 │ 10. git push                                                 │
 │ 11. exit → Ralph loop spawns a new Claude in a new container │
 └──────────────────────────────────────────────────────────────┘
```

### Гонка и почему побеждает git

Два агента просыпаются одновременно. Оба смотрят на открытые задачи. Оба решают, что самое очевидное — «починить segfault в codegen/x86.rs». Оба пишут `current_tasks/fix-segfault-x86.txt`. Оба `git push`.

Один push побеждает. Другой отклоняется — `non-fast-forward`. Этот агент делает `git pull --rebase`, видит, что lock уже существует, понимает, что опоздал, и идёт выбирать другую задачу.

Это ход: **модель конкурентности git — это и модель конкурентности команды агентов.** Lock'и, слияния и разрешение конфликтов — всё бесплатно, потому что git уже это решает. Вы не строите paper по распределённым системам поверх LLM; Вы наследуете один из 2005 года.

## Принцип проектирования 1: пишите тесты для Claude, а не для себя

Это урок, который я бы втатуировал на лбу любого, кто строит харнесы для агентов.

Когда человек читает падающий тест, он бесплатно получает многое: может пробежать глазами 500 строк вывода, заметить подозрительный паттерн, держать гипотезу в голове и пойти отлаживать. LLM в цикле не может хорошо делать ничего из этого. Ваш тестовый харнес должен быть спроектирован под его ограничения, а не под Ваши.

### Загрязнение контекстного окна

Дефолтное поведение большинства тестовых раннеров — «выгрузить всё в stdout». Это нормально для человека, листающего в терминале; это катастрофа для Claude. Тысяча строк вывода тестов сожрёт контекстное окно, и к тому моменту, когда Claude будет рассуждать об ошибке, он забудёт промпт.

**Решение:** сделайте тестовый харнес максимально лаконичным по умолчанию, а подробности отправляйте в файлы.

```python
def run_test(name: str) -> TestResult:
    result = execute(name)
    # Human-visible (and LLM-visible) output: one line.
    if result.ok:
        print(f"PASS {name}")
    else:
        print(f"ERROR {name}  see logs/{name}.log")
    # Everything else goes to disk.
    write(f"logs/{name}.log", result.full_output)
    return result
```

Теперь Claude видит:

```
PASS  tests/struct_layout_01
PASS  tests/struct_layout_02
ERROR tests/struct_layout_03  see logs/struct_layout_03.log
PASS  tests/struct_layout_04
```

Claude может сделать `grep ERROR` по всему запуску, выбрать одну ошибку, затем `cat` только релевантного лога. Контекст остаётся маленьким. Внимание остаётся сфокусированным.

### Маркер `ERROR`

Заметьте литерал `ERROR`. Это намеренно. Согласованный, grep'аемый маркер — один из самых высокодоходных соглашений в любом харнесе для агента. Он означает, что Claude может навигировать по тысячам результатов тестов одним `grep -l ERROR logs/`. Не пишите «failed» в одном месте и «FAIL» в другом и «❌» в третьем. Выберите один. Поставьте его везде.

### Слепота ко времени

Claude нет часов реального времени. Он не может отличить «этот тест медленный» от «этот тест завис». Если Вы позволяете агенту запускать полный набор тестов каждую сессию, половина Ваших агентов будет сидеть 40 минут, решит, что ничего не происходит, убьёт набор и попробует какое-нибудь безумное рефакторинг.

**Решение:** добавьте подрежим `--fast`, запускающий 1–10% тестов, выбираемых детерминированно для каждого агента:

```python
def select_tests(all_tests: list[str], agent_id: str, fraction=0.05):
    # Same agent → same subsample within a session.
    # Different agents → different subsamples, so the fleet covers
    # more ground than any single agent does.
    rng = seeded_random(agent_id)
    k = max(1, int(len(all_tests) * fraction))
    return rng.sample(all_tests, k)
```

Два свойства важны здесь:

- **Детерминированность для каждого агента**: тот же агент, запускающий ту же команду дважды, видит одни и те же результаты. Иначе Claude не может рассуждать о том, исправил ли его изменение что-то.
- **Различность внутри флота**: каждый агент исследует другой угол тестового набора, и объединение их покрытия намного больше, чем у любого одного агента.

### Файлы прогресса и README

Единственный долгоживущий кусок памяти — файловая система. Агенты Carlini поддерживают:

- `README.md` в вершине каждого модуля: *что делает, что работает, что не работает*.
- `PROGRESS.md` и файлы прогресса по областям: *что пробовалось, что упало, какая текущая теория*.

Это не для людей. Это для следующего Claude, который проснётся. Когда Вы рассматриваете файлы прогресса как **сообщения будущему «я»**, они становятся самым ценным артефактом в репозитории.

## Принцип проектирования 2: удобный для параллелизма декомпозиция задач

Параллелизм — не факт о Вашей команде. Это факт о Вашей *работе*.

### Лёгкий режим: множество независимых ошибок

Счастливый случай для команды агентов — тестовый набор с множеством независимо падающих тестов. 200 падающих тестов, 16 агентов, каждый хватает один, чинит, коммитит, переходит дальше. Минимум конфликтов. Линейное масштабирование. Все уходят домой довольные.

```
┌───────────────────────┐
│  200 failing tests    │
├───────────────────────┤
│  agent-1 → test 47    │
│  agent-2 → test 113   │
│  agent-3 → test 2     │
│  agent-4 → test 88    │
│   ...                 │
└───────────────────────┘
```

### Тяжёлый режим: один монолитный билд

Теперь представьте цель: *скомпилировать Linux-ядро*. Одна задача. Один билд. Один сломанный артефакт. 16 агентов начинают читать исходники ядра и предлагать исправления. Они перезаписывают друг друга. Они не согласны, чинить ли баг в парсере или в codegen. Репозиторий превращается в derby на уничтожение.

Больше агентов теперь *хуже*. Бросать больше Claude на одну неделимую проблему — это способ потратить $20K за полдня без результата.

Решение — не «добавить оркестратор». Решение — **декомпозировать монолит на независимые ошибки**. Что приводит нас к самой умной идее во всём проекте.

## GCC как оракул: бисекция ядра

Когда команда попыталась скомпилировать Linux-ядро своим новым компилятором, вещи сломались вspectacular, запутанных формах. Сотни файлов вносили вклад в один сломанный образ. Ни Claude, ни человек не могли посмотреть на «ядро не загрузилось» и сказать, какой из 20 000 C-файлов обнажил баг.

Ход: **используйте GCC как известно-работающий оракул и позвольте Claude компилировать только подмножество**.

```
┌────────────────────────────────────────────────────────────┐
│  For each kernel .c file:                                  │
│    flip a coin biased toward GCC (say 95/5)                │
│    → if GCC: compile with GCC (trusted)                    │
│    → if Claude: compile with Claude's compiler             │
│                                                            │
│  Link everything into a kernel image. Boot it.             │
│                                                            │
│  If it boots: great. Try a larger Claude subset.           │
│  If it breaks: Claude's subset contains the bug.           │
│                 → bisect the subset, narrow down,          │
│                   identify the specific failing file,      │
│                   file a focused bug report,               │
│                   add it as an isolated failing test.      │
└────────────────────────────────────────────────────────────┘
```

Псевдокод:

```python
def locate_bad_file(c_files: list[str]) -> str:
    # Standard bisection against a known-good oracle.
    lo, hi = 0, len(c_files)
    while hi - lo > 1:
        mid = (lo + hi) // 2
        subset = c_files[lo:mid]
        image = build_kernel(subset, compiler="claude",
                             rest_compiler="gcc")
        if boots(image):
            lo = mid       # bug is in the upper half
        else:
            hi = mid       # bug is in the lower half
    return c_files[lo]
```

Когда Вы локализовали баг в одном файле, Вы сделали критический шаг: превратили монолитную ошибку в **независимый, воспроизводимый тест-кейс**. Теперь Вы снова в лёгком режиме — конкретный падающий тест, который может взять на себя один Claude.

Общая форма этого паттерна:

> **Когда задача слишком большая для параллелизации, найдите доверенный оракул, пропустите большую часть нагрузки через оракул и позвольте агентам атаковать уменьшающееся подмножество, пока каждая оставшаяся ошибка не станет локальной.**

Это паттерн проектирования харнеса, а не компилятора. Он применим ко всему, где у Вас есть монолитная система с доверенным эталоном — пайплайны данных, подкомпоненты обучения моделей, генерация конфигов, даже рендеринг UI.

## Специализация ролей агентов

Все 16 агентов — тот же бинарник Claude, но их промпты дают им разные *работы*. Это звучит как мелочь; это не так. Назначение каждому агенту уникальной роли резко снижает патологию «все толпуются вокруг одной ошибки».

Пять ролей из проекта, перефразированных:

```
┌──────────────────────────┬──────────────────────────────────┐
│ Role                     │ Prompt flavor                    │
├──────────────────────────┼──────────────────────────────────┤
│ duplicate-code-cleaner   │ Find repeated patterns, extract  │
│                          │ helpers, reduce LOC without      │
│                          │ changing behavior.               │
├──────────────────────────┼──────────────────────────────────┤
│ performance-tuner        │ Profile, find hot paths, apply   │
│                          │ local optimizations. Do not      │
│                          │ change semantics.                │
├──────────────────────────┼──────────────────────────────────┤
│ codegen-optimizer        │ Focus on the backend. Improve    │
│                          │ emitted assembly quality.        │
├──────────────────────────┼──────────────────────────────────┤
│ design-critic            │ Do not write code. Read the repo │
│                          │ and file TODOs for design smell, │
│                          │ dead code, inconsistency.        │
├──────────────────────────┼──────────────────────────────────┤
│ doc-maintainer           │ Keep README / PROGRESS files     │
│                          │ honest and current.              │
└──────────────────────────┴──────────────────────────────────┘
```

Разделение ролей важно, потому что оно **изменяет то, какие задачи агент считает «самыми очевидными».** `design-critic`, смотрящий на репозиторий, видит другие «следующие работы», чем `performance-tuner`, смотрящий на тот же репозиторий. Флот покрывает больше рабочей поверхности без центрального планировщика, решающего, кто что делает.

Для Вашего собственного харнеса общий принцип: **если два агента постоянно сталкиваются, обычно потому, что их промпты не различают то, как выглядит «очевидный следующий шаг».**

## Когда НЕ строить команду агентов

Честная часть статьи. Команды агентов дорогие, неловкие и иногда волшебные. Они не подходят для большинства работ. **Не** стройте команду, если верно хотя бы одно:

- **Проект мал.** Если один сфокусированный Claude может завершить задачу за один проход, флот из 16 создаст накладные расходы координации, конфликты слияний и счёт на $500 за задачу на $5. Один острый нож бьёт шестнадцать тупых.
- **Работа требует тесной координации.** Дизайны с множеством сквозных решений — API-схемы, протокольные инварианты, общие модели данных — уничтожаются параллельными агентами. Агенты хороши в «почини этот падающий тест». Они плохи в «давайте договоримся об иерархии типов ошибок».
- **У Вас нет хорошего тестового оракула.** У компилятора есть GCC torture test suite и ядро, которое либо загружается, либо нет. У большинства ПО нет доверенного, автоматизируемого, детерминированного сигнала «оно работает?». Без него Ваши агенты оптимизируют под то, что проверяет верификатор, а не под то, что Вы хотите на самом деле — а поскольку верификатор несовершенен, они оптимизируют не под то.
- **Вы не можете дёшево перезапускать.** Ralph loop предполагает, что перезапуск бесплатен. Если Вашим агентам нужно 30 минут прогреваться перед полезной работой, цикл амортизируется плохо, и флот проигрывает одному долгоживущему процессу.

Если хотя бы одно из этого верно, используйте один Claude с хорошим промптом и остановитесь.

## Чему это учит строителей харнесов

Отступая от конкретного проекта, пять идей чисто переносятся:

1. **Git — примитив координации агентов.** Не стройте очередь задач, сервис блокировок, шину сообщений. Используйте git. Lock'и, слияния, разрешение конфликтов, история, откат — всё бесплатно, и оно спроектировано людьми, которые уже тщательно думали о конкурентности.
2. **Качество тестового набора — потолок способностей агента.** Claude может быть только настолько хорош, насколько хорош сигнал, который Вы ему даёте. Непропорционально инвестируйте в тесты, оракулов и верификаторы. Средний агент с отличным тестовым набором бьёт отличного агента со средним тестовым набором каждый раз.
3. **Проектируйте харнес для Claude, а не для себя.** Лаконичный вывод по умолчанию. Маркеры `ERROR`. Полные логи в файлы. Детерминированная подвыборка. Файлы прогресса как письма будущим сессиям. Всё это существует потому, что агент отличается от вас — более забывчив, более подвержен отвлечениям, слеп ко времени.
4. **Декомпозируйте монолиты через известно-работающие оракулы.** Когда задача сопротивляется параллелизму, не добавляйте агентов; добавьте доверенный эталон, пропустите большую часть работы через него и позвольте агентам атаковать уменьшающийся остаток. GCC как оракул — общий паттерн.
5. **Специализируйте через промпты.** Та же модель, разная работа. Самый дешёвый и масштабируемый способ не давать 16 агентам наступать друг другу на ноги — дать им разные определения «очевидного следующего шага».

Счёт в $20 000 оплатил компилятор. Бесплатный бонус — чертёж того, как работают команды агентов, когда они работают.
