# Защитные ограничения

> **Ключевая идея:** Агент без защитных ограничений — это угроза. Модель будет делать ровно то, что ей скажут — включая то, что ей скажет prompt-injection. Защитные ограничения — это слой разрешений между «модель хочет сделать X» и «харнес действительно делает X».

## Зачем нужны защитные ограничения

Модель генерирует текст. Этот текст включает вызовы инструментов. Харнес выполняет эти вызовы. Это значит, что всё, что влияет на вывод модели, может повлиять на то, что делает харнес — включая вредоносный контент в файлах, веб-страницах или сообщениях пользователя.

Это **prompt-injection**: атакующий встраивает инструкции в данные, которые читает агент, и модель следует этим инструкциям вместо исходной задачи. Без защитных ограничений prompt-injection может:

- Удалить файлы (`rm -rf /`)
- Утечь переменные окружения (API-ключи, токены)
- Выполнить произвольный код на хосте
- Отправить несанкционированные сообщения от имени пользователя

Защитные ограничения делают харнес финальной инстанцией в том, какие действия допустимы, независимо от того, что запрашивает модель.

## Модель границы доверия

У каждого харнеса есть граница доверия между моделью и средой выполнения. Харнес опосредует все переходы:

```
┌──────────────────────────────────┐
│           MODEL SPACE            │
│  (reasoning, tool call requests) │
└────────────┬─────────────────────┘
             │ tool call request
             ▼
┌──────────────────────────────────┐
│         GUARDRAIL LAYER          │
│  Permission check → Allow/Deny   │
└────────────┬─────────────────────┘
             │ approved call
             ▼
┌──────────────────────────────────┐
│        EXECUTION SPACE           │
│  (filesystem, network, shell)    │
└──────────────────────────────────┘
```

Слой защитных ограничений перехватывает каждый вызов инструмента перед выполнением. Он может:
- **Разрешить** — выполнить как запрошено
- **Запретить** — вернуть ошибку модели
- **Модифицировать** — перезаписать вызов (например, ограничить путь к файлу безопасной директорией)
- **Запросить** — спросить разрешения у человека перед продолжением

## Модели разрешений

### Белый список (самые строгие)

Только явно разрешённые действия допустимы. Всё остальное запрещено по умолчанию:

```python
ALLOWED_TOOLS = {
    "read_file": {"paths": ["/workspace/**"]},
    "write_file": {"paths": ["/workspace/**"]},
    "run_command": {"commands": ["npm test", "npm run build"]},
}

def check_permission(tool_name: str, args: dict) -> bool:
    if tool_name not in ALLOWED_TOOLS:
        return False
    policy = ALLOWED_TOOLS[tool_name]
    if "paths" in policy:
        return any(fnmatch(args.get("path", ""), p) for p in policy["paths"])
    if "commands" in policy:
        return args.get("command") in policy["commands"]
    return True
```

### Чёрный список (пермиссивный)

Всё разрешено, кроме явно заблокированных действий:

```python
BLOCKED_PATTERNS = [
    (r"rm\s+-rf\s+/", "Refusing to delete root filesystem"),
    (r"curl.*\|\s*sh", "Refusing to pipe remote script to shell"),
    (r"env\s+|printenv|echo\s+\$", "Refusing to expose environment variables"),
]

def check_command(command: str) -> tuple[bool, str]:
    for pattern, reason in BLOCKED_PATTERNS:
        if re.search(pattern, command):
            return False, reason
    return True, ""
```

### Каскадное одобрение

Разные уровни риска запускают разные процессы одобрения:

| Уровень риска | Примеры | Действие |
|---------------|---------|----------|
| **Низкий** | Чтение файлов, поиск | Автоодобрение |
| **Средний** | Запись файлов, запуск тестов | Автоодобрение с логированием |
| **Высокий** | Выполнение shell-команд, сетевые запросы | Требуется одобрение человека |
| **Критический** | Удаление файлов, push в git, отправка сообщений | Всегда требуется явное одобрение |

```python
def get_risk_level(tool_name: str, args: dict) -> str:
    if tool_name == "read_file":
        return "low"
    if tool_name == "write_file":
        return "medium"
    if tool_name == "run_command":
        cmd = args.get("command", "")
        if any(k in cmd for k in ["rm", "git push", "curl"]):
            return "critical"
        return "high"
    return "medium"
```

## Песочница

Защитные ограничения обеспечивают политику; песочницы обеспечивают изоляцию. Песочница — это ограниченная среда выполнения, которая ограничивает возможности кода на уровне ОС:

| Технология | Уровень изоляции | Накладные расходы | Применение |
|------------|-----------------|-------------------|------------|
| **chroot** | Только файловая система | Минимальные | Базовое ограничение путей |
| **Docker** | Процесс + файловая система + сеть | Низкие | Разработка, CI/CD |
| **Firecracker microVM** | Полная ВМ | Средние | Production, мультитенантность |
| **gVisor** | Уровень системных вызовов | Низкие–средние | Высоконагруженные задачи |
| **WASM** | Уровень языка | Минимальные | Браузерные агенты |

Большинство production-харнесов используют Docker для разработки и Firecracker (или аналог) для продакшена. Ключевой принцип: **код агента при выполнении никогда не должен иметь доступа к файловой системе, сети или пространству процессов хоста**.

## Очистка входных данных

Помимо защитных ограничений на уровне инструментов, харнес должен очищать входные данные для снижения риска prompt-injection:

```python
def sanitize_tool_result(result: str, max_length: int = 50_000) -> str:
    """Truncate and mark external content as untrusted."""
    if len(result) > max_length:
        result = result[:max_length] + "\n[TRUNCATED]"
    # Wrap in markers so the model knows this is external data
    return f"<tool_result>\n{result}\n</tool_result>"
```

Контент из внешних источников (веб-страницы, загруженные пользователем файлы, ответы API) должен быть чётко обозначен в контексте, чтобы модель могла отличить инструкции от данных.

## Частые ошибки

- **Нет защитных ограничений вообще** — по умолчанию для большинства любительских харнесов. Нормально для локальной разработки, катастрофично для продакшена.
- **Защитные ограничения только в промпте** — сказать модели «не удаляй файлы» — это не защитное ограничение. Модель может быть переопределена через prompt-injection. Настоящие защитные ограничения обеспечиваются кодом, а не текстом.
- **Слишком строгие разрешения** — агент, который ничего полезного не может сделать, не будет использоваться. Балансируйте безопасность и полезность.
- **Нет логирования заблокированных действий** — понимание того, что агент *пытался* сделать, но было заблокировано, критически важно для отладки и улучшения промптов.
