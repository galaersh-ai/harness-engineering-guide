# Оркестрация мультиагентов

> **Ключевая идея:** Одиночный агент упирается в жёсткие лимиты — размер контекстного окна, предметная специализация и последовательное выполнение. Оркестрация мультиагентов разбивает работу на несколько агентов, каждый со своим контекстом и инструментами, координируемых паттернами, заимствованными из распределённых систем.

Статья [Субагент](sub-agent.md) охватывает простейший случай: один лидер, порождающий исполнителей для параллельных задач. Эта статья идёт дальше — паттерны оркестрации для систем, где несколько агентов сотрудничают, специализируются и масштабируются.

## Зачем мультиагентность?

Одиночный агент, выполняющий один агентный цикл, — это стандарт. Он хорошо работает для большинства задач. Но три стены в итоге вынуждают выйти за его пределы:

### 1. Ограничения контекстного окна

Даже с 200K-токенными окнами сложный проект может исчерпать доступный контекст. Рефакторинг кодовой базы, затрагивающий 80 файлов, исследовательская задача, потребляющая 40 исходных документов, многоэтапный пайплайн, накапливающий выводы инструментов — всё это может переполнить одно окно. Когда контекст заполняется, модель начинает забывать ранние инструкции и терять связность.

Мультиагентность разбивает работу. Каждый агент получает свежее контекстное окно, посвящённое его части задачи.

### 2. Специализация

Универсальный агент, который пишет код, рецензирует PR, генерирует маркетинговый текст и проводит анализ данных, mediocre во всём. Агенты работают лучше, когда их системный промпт, инструменты и контекст настроены под конкретную предметную область. «Агент-рецензент кода» с загруженными инструментами линтинга и руководствами по стилю выдаёт лучшие рецензии, чем универсальный агент, который случайно тоже имеет доступ к линтеру.

### 3. Параллелизм

Одиночный агент последователен. Он вызывает один инструмент, ждёт результат, рассуждает, вызывает другой инструмент. Когда у Вас пять независимых задач — написать тесты для пяти модулей, исследовать пять конкурентов, перевести на пять языков — один агент обрабатывает их по очереди. Пять агентов обрабатывают их одновременно.

## Паттерны оркестрации

Четыре паттерна охватывают большинство мультиагентных архитектур. Они не взаимоисключающие — production-системы часто комбинируют их.

### Последовательный пайплайн

Агент A завершает работу, передаёт вывод агенту B, который передаёт свой вывод агенту C. Каждый агент трансформирует или обогащает результат.

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ Ingest  │────►│ Analyze │────►│  Draft  │────►│ Review  │
│  Agent  │     │  Agent  │     │  Agent  │     │  Agent  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

**Когда использовать:** Задачи с естественными этапами, где каждый этап требует разных экспертизы или инструментов. Пример: сбор данных → очистка и анализ → генерация отчёта → вычитка и факт-чекинг.

**Ключевое свойство:** Каждый агент видит только вывод предыдущего этапа, а не полный накопленный контекст. Это преимущество — оно обеспечивает чёткие интерфейсы между этапами и предотвращает раздувание контекста.

### Развёртывание (Fan-Out / Fan-In)

Координатор отправляет одни и те же (или связанные) задачи N агентам параллельно, затем собирает и объединяет все результаты.

```
                    ┌───────────┐
               ┌───►│ Worker A  │───┐
               │    └───────────┘   │
┌──────────┐   │    ┌───────────┐   │    ┌──────────┐
│Dispatcher│───┼───►│ Worker B  │───┼───►│  Merger  │
└──────────┘   │    └───────────┘   │    └──────────┘
               │    ┌───────────┐   │
               └───►│ Worker C  │───┘
                    └───────────┘
```

**Когда использовать:** Задачи, которые декомпозируются на независимые, параллелизуемые подзадачи. Пример: перевод документа на 8 языков, написание модульных тестов для 10 модулей, исследование 5 конкурентных продуктов.

**Ключевое свойство:** Исполнители не взаимодействуют друг с другом. Агент-объединитель (или сам диспетчер) отвечает за комбинирование результатов и разрешение конфликтов.

### Супервайзер

Один надзорный агент принимает все решения о делегировании. Он наблюдает текущее состояние, решает, какого специалиста вызвать следующим, читает результат и решает следующий шаг. В отличие от пайплайна, супервайзер может зацикливаться, переделегировать или вызвать того же специалиста несколько раз на основе промежуточных результатов.

```
                 ┌─────────────────┐
                 │   Supervisor    │
                 │ (decides, loops)│
                 └───┬───┬───┬────┘
                     │   │   │
            ┌────────┘   │   └────────┐
            ▼            ▼            ▼
      ┌──────────┐ ┌──────────┐ ┌──────────┐
      │ Code     │ │ Research │ │ Review   │
      │ Agent    │ │ Agent    │ │ Agent    │
      └──────────┘ └──────────┘ └──────────┘
```

**Когда использовать:** Задачи, где следующий шаг зависит от результата предыдущего, и решение *какого* специалиста вызвать требует суждения. Пример: coding-супервайзер, делегирующий реализацию code-агенту, отправляющий результат агенту-рецензенту и возвращающийся к code-агенту, если рецензия находит проблемы.

**Ключевое свойство:** Супервайзер поддерживает общий план и состояние. Специалисты бесостоятельны — получают задачу, возвращают результат и забывают. Это держит контекстные окна специалистов чистыми.

### Равный-равному (Peer-to-Peer)

Агенты общаются напрямую друг с другом без центрального координатора. У каждого агента свой почтовый ящик, и он может отправлять сообщения любому другому агенту.

**Когда использовать:** Симуляции, дискуссионные рассуждения (агент A аргументирует «за», агент B «против», агент C судит), или системы, где агенты представляют различных стейкхолдеров.

**Ключевое свойство:** Нет единой точки координации. Это делает peer-to-peer самым гибким паттерном, но и самым сложным для отладки. Без супервайзера сложно гарантировать завершение или отслеживать прогресс.

**Предупреждение:** Peer-to-peer редко является правильным выбором в production-системах харнесов. Отсутствие центрального управления делает обработку ошибок, обеспечение таймаутов и сбор результатов значительно сложнее. Предпочитайте супервайзер или fan-out/fan-in, если у Вас нет конкретной причины для децентрализованной коммуникации.

## Реализация в харнесе

Харнес (хост-программа, оборачивающая LLM в работающего агента — см. [Что такое харнес](what-is-harness.md)) реализует оркестрацию мультиагентов через четыре механизма: порождение субагентов, изоляция контекста, коммуникация «родитель читает результаты детей» и обработка таймаутов.

### Порождение субагентов

Базовый примитив — порождение изолированной сессии агента. В OpenClaw это паттерн `sessions_spawn`:

```
Parent Agent
    │
    ├── sessions_spawn(label="research", task="Find competitor pricing")
    │       → creates isolated session with own context, tools, model
    │
    ├── sessions_spawn(label="writer", task="Draft the pricing page")
    │       → another isolated session
    │
    └── waits for both to complete (push-based)
```

Каждая порождённая сессия — полностью независимый агент со своим:
- Контекстным окном (нет общего бюджета токенов с родителем)
- Системным промптом (может быть настроен для каждого субагента)
- Набором инструментов (может быть ограничен или расширен)
- Моделью (может использовать более дешёвую модель для простых задач)

Родитель указывает метку, описание задачи и опционально модель или уровень мышления. Харнес управляет жизненным циклом процесса, таймаутом и доставкой результатов.

### Изоляция контекста

Изоляция контекста — не удобство, а ключевое ограничение проектирования. Контекстное окно каждого субагента полностью независимо:

```
┌──────────────────────┐    ┌──────────────────────┐
│   Parent Context     │    │  Sub-Agent Context    │
│                      │    │                       │
│ [system prompt]      │    │ [system prompt]       │
│ [user conversation]  │    │ [task instruction]    │
│ [tool results A,B,C] │    │ [tool results X,Y]   │
│ [128K tokens used]   │    │ [12K tokens used]     │
│                      │    │                       │
│  Cannot see ────────►│ X  │◄──── Cannot see       │
│  sub-agent context   │    │  parent context       │
└──────────────────────┘    └──────────────────────┘
```

**Почему это важно:** Родительский агент с 128K токенами накопленного диалога может породить субагента, который начинает со свежего 200K бюджета токенов. Субагент не обременён нерелевантной историей — он видит только свою инструкцию задачи и тот контекст, который родитель явно передаёт.

Следствие: субагенты не могут читать переменные родителя, результаты инструментов или историю диалога. Вся передача информации явная — родитель включает релевантный контекст в строку инструкции задачи.

### Коммуникация: родитель читает результаты детей

Коммуникация мультиагентов в харнесе следует строгому паттерну: **родитель читает результаты детей; дети не пишут в память родителя.**

```
1. Parent spawns child with task instruction (one-way write)
2. Child executes independently
3. Child completes → Harness delivers result to parent (push-based)
4. Parent reads the result in its next Agentic Loop iteration
```

Нет общей памяти, нет шины сообщений, нет механизма обратных вызовов, где ребёнок модифицирует состояние родителя. Это ограничение резко упрощает рассуждения о системе — родитель всегда является единственным источником правды о общем прогрессе.

В OpenClaw завершение **push-based**: когда субагент завершается, результат автоматически объявляется в сессии родителя. Родителю не нужно опрашивать. Это устраняет самую распространённую ошибку мультиагентов: частые циклы опроса, тратящие токены на проверку «он уже готов?».

### Таймаут и обработка ошибок

Субагенты могут зависать, падать или выдавать мусор. Production-харнес должен обрабатывать все три варианта:

```python
@dataclass
class SubAgentConfig:
    task: str
    label: str
    timeout_seconds: int = 300       # Kill after 5 minutes
    max_retries: int = 1             # Retry once on failure
    fallback: str | None = None      # Alternative task on repeated failure

class SubAgentManager:
    def spawn_with_safeguards(self, config: SubAgentConfig) -> str:
        for attempt in range(config.max_retries + 1):
            try:
                result = self.spawn(
                    task=config.task,
                    label=config.label,
                    timeout=config.timeout_seconds,
                )
                if self.validate_result(result):
                    return result
                # Invalid result — retry with clarified instructions
                config.task += "\n\nPrevious attempt produced invalid output. Be precise."
            except TimeoutError:
                if attempt == config.max_retries and config.fallback:
                    return self.spawn(task=config.fallback, label=f"{config.label}-fallback")
                continue
            except Exception as e:
                log.error(f"Sub-agent {config.label} failed: {e}")
                if attempt == config.max_retries:
                    return f"[FAILED] {config.label}: {e}"
        return f"[FAILED] {config.label}: max retries exceeded"
```

**Ключевые принципы:**
- **Всегда устанавливайте таймауты.** Субагент без таймаута — это утечка ресурсов, которая ждёт своего часа.
- **Проверяйте результаты.** Субагент, который «завершается» с пустым выводом или сообщением об ошибке — это не успех.
- **Деградируйте корректно.** Когда субагент падает, родитель должен продолжить с частичными результатами, а не прерываться полностью — если только упавшая задача не критична.
- **Ограничивайте глубину.** Субагенты, порождающие своих субагентов, создают экспоненциальную сложность. Ограничьте рекурсию 1–2 уровнями.

## Примеры из реальной жизни

Три open-source проекта демонстрируют оркестрацию мультиагентов в разных масштабах.

### Multica (14K+ Stars)

**Паттерн:** Супервайзер + Fan-Out / Fan-In

Multica рассматривает агентов как членов команды на доске проекта. Человек (или ведущий агент) назначает GitHub-issues исполнителям-агентам, каждый работает независимо. Система отслеживает прогресс на доске в стиле Kanban, показывая, какой агент работает над чем, что на рецензии и что смержено.

Ключевые решения проектирования:
- **Многорантаймная поддержка** — агенты могут работать на Claude Code, Codex, OpenClaw или OpenCode. Оркестратору всё равно, какой рантайм использует исполнитель, главное — чтобы он выдавал ожидаемый вывод.
- **Комбинированные навыки** — агенты могут комбинировать несколько навыков (например, агент «исправь и протестируй» загружает и навык программирования, и навык тестирования).
- **Делегирование через issues** — работа организована вокруг GitHub-issues, а не абстрактных «задач». Это даёт каждому агенту конкретную, отслеживаемую единицу работы с чёткими критериями приёмки.

### Paseo (3.6K+ Stars)

**Паттерн:** Супервайзер с кросс-устройственным выполнением

Paseo предоставляет единый унифицированный интерфейс для взаимодействия с несколькими агентами. Вы отдаёте команду, и Paseo маршрутизирует её нужному агенту — или развёртывает её на несколько агентов для параллельного выполнения.

Ключевые решения проектирования:
- **Один интерфейс, много агентов** — пользователю не нужно знать, какой агент обрабатывает какую задачу. Маршрутизационный слой Paseo решает сам.
- **Кросс-устройственно** — та же оркестрация работает с телефона, десктопного приложения или CLI. Агенты работают на сервере; интерфейс — просто тонкий клиент.
- **Параллелизм по умолчанию** — когда задачи независимы, Paseo автоматически параллелизирует между доступными агентами вместо постановки в очередь.

### OpenClaw

**Паттерн:** Нативное порождение субагентов с push-завершением

OpenClaw реализует оркестрацию мультиагентов как примитив первого класса через `sessions_spawn`. Родительский агент порождает субагентов одним вызовом инструмента, указывая метку, задачу и опциональную конфигурацию модели/мышления.

Ключевые решения проектирования:
- **Push-завершение** — когда субагент завершается, его результат автоматически объявляется в родительской сессии. Опрос не нужен. Это явно документировано: *«Completion is push-based: it will auto-announce when done.»*
- **Отслеживание по меткам** — каждый субагент получает человекочитаемую метку (например, `article-research`, `code-review`). Родитель может перечислять, направлять или убивать субагентов по метке.
- **Ограничение глубины** — субагенты могут быть настроены с максимальной глубиной для предотвращения неконтролируемой рекурсивной генерации.
- **Инжекция контекста** — родитель может инжектировать конкретный контекст (файлы, инструкции, ограничения) в сессию субагента, не разделяя всю историю диалога.

## Когда НЕ использовать мультиагентность

Оркестрация мультиагентов имеет реальную стоимость: увеличенная задержка (накладные расходы на порождение), повышенное потребление токенов (дублирование системных промптов и контекста между агентами) и добавленная сложность отладки. Избегайте её, когда:

### Задача помещается в одно контекстное окно

Если задача требует 10K токенов контекста и 5 вызовов инструментов, один агент обработает её быстрее и надёжнее, чем порождение исполнителей. Накладные расходы мультиагентной координации (порождение, передача контекста, объединение результатов) превышают выгоду.

### Стоимость координации превышает выгоду

Задача, требующая тесного обмена между двумя агентами — где агенту A нужен результат агента B для следующего шага, и наоборот — лучше обрабатывается одним агентом, делающим последовательные вызовы инструментов. Мультиагентность блестит, когда работа *независима*; она страдает, когда работа глубоко переплетена.

### Требуется строгий порядок

Если шаги должны выполняться в точном порядке с точными передачами, и нет возможности параллелизации, один агент с пошаговым планом проще и детерминированнее. Последовательный пайплайн агентов добавляет задержку на каждой границе этапа без выигрыша в параллелизме — используйте его только когда этапы действительно выигрывают от изоляции контекста или специализации.

## Антипаттерны

### Общее изменяемое состояние

Два агента, одновременно пишущих в один файл, строку базы данных или переменную. Это создаёт гонки, перезаписанную работу и недетерминированные результаты. Исправление: закрепите за каждым агентом владение отдельными ресурсами. Если два агента должны вносить вклад в один артефакт, используйте паттерн fan-out/fan-in, где агент-объединитель последовательно комбинирует их выводы.

### Неограниченное развёртывание

Порождение агентов пропорционально размеру входных данных без лимита. «Переведите этот документ на все 50 поддерживаемых языков» порождает 50 одновременных агентов, перегружая систему. Исправление: установите лимит `max_workers` и обрабатывайте пачками.

```python
# Bad: unbounded
for lang in all_50_languages:
    spawn_agent(task=f"Translate to {lang}")

# Good: bounded batches
BATCH_SIZE = 8
for batch in chunked(all_50_languages, BATCH_SIZE):
    results = fan_out_fan_in(
        [f"Translate to {lang}" for lang in batch],
        merge_prompt="Collect all translations.",
    )
```

### Нет таймаута или автоматического выключателя

Субагент, который зависает — ожидает API с лимитом, застрял в бесконечном цикле или просто выполняет сложную задачу — блокирует всю оркестрацию, если нет таймаута. Каждое порождение субагента должно включать:
- **Таймаут**, убивающий агента после максимальной длительности
- **Автоматический выключатель**, прекращающий повторы после N неудач
- **Путь деградации**, позволяющий родителю продолжить с частичными результатами

### Чрезмерная декомпозиция

Разбиение 30-секундной задачи на 5 субагентов, каждый тратящий 15 секунд на порождение и возврат результатов. Общее время выполнения увеличивается с 30 до 75 секунд. Мультиагентность — инструмент управления сложностью и параллелизмом, а не стандарт для каждой задачи.

## Выбор правильного паттерна

| Ситуация | Паттерн | Причина |
|----------|---------|---------|
| Этапы с разной экспертизой | Последовательный пайплайн | Каждый этап получает специализированный контекст и инструменты |
| N независимых одинаковых задач | Fan-Out / Fan-In | Максимальный параллелизм, простое объединение |
| Сложная задача, требующая суждения о следующих шагах | Супервайзер | Центральное принятие решений, гибкая маршрутизация |
| Дискуссия или мультиперспективный анализ | Peer-to-Peer | Агенты представляют разные точки зрения |
| Смесь параллельной и последовательной работы | Супервайзер + Fan-Out | Супервайзер решает, когда параллелизировать |

Начните с простейшего паттерна, который работает. Один агент с хорошим использованием инструментов обрабатывает 80% задач. Когда Вы упрётесь в ограничения, описанные в разделе [Зачем мультиагентность?](#почему-мультиагентность), возьмите самый лёгкий паттерн оркестрации, решающий конкретное узкое место. Статья [Субагент](sub-agent.md) охватывает простейший паттерн делегирования — начните с него, прежде чем строить полный слой оркестрации.
