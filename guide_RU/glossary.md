# Глоссарий

Ключевые термины, используемые в руководстве по инженерии харнесов, объяснённые простым языком.

---

**Харнес (Harness)**
Код, который оборачивает ИИ-модель и превращает её в функционального агента. Управляет циклом инструментов, контекстом, памятью и потоком решений. Без харнеса у Вас есть чатбот; с ним — агент.

**Среда выполнения (Runtime)**
Среда, в которой работает харнес. Включает операционную систему, файловую систему, сетевой доступ и песочницу. Среда выполнения определяет, что агент может физически *делать* — харнес определяет, что он *выбирает* делать.

**Фреймворк (Framework)**
Библиотека или SDK для построения харнесов. Фреймворки предоставляют абстракции (реестры инструментов, хранилища памяти, шаблоны промптов), чтобы Вам не приходилось строить всё с нуля. Примеры: LangChain, CrewAI, AutoGen. Фреймворк — это набор инструментов; харнес — готовый продукт.

**Контекстное окно (Context Window)**
Максимальный объём текста (в токенах), который модель может обработать за один запрос. Обычно 128K–200K токенов для моделей нового поколения. Всё, что агент «знает» за ход — инструкции, память, история диалога, содержимое файлов — должно поместиться в это окно.

**Цикл инструментов (Tool Loop)**
Базовый цикл выполнения агента: вызов модели → модель запрашивает инструмент → выполнение инструмента → возврат результата → повтор. Цикл продолжается, пока модель не выдаст финальный текстовый ответ без вызовов инструментов. Каждый харнес реализует какую-то версию этого цикла.

**Навык (Skill)**
Автономный пакет возможностей, расширяющий способности агента. Состоит из инструкции SKILL.md и вспомогательного кода или шаблонов. Навыки — это «толстая» часть паттерна «тонкий харнес, толстые навыки» — они кодируют предметные знания, чтобы харнес оставался универсальным.

**AGENTS.md**
Файл конфигурации, определяющий поведение агента в рабочем пространстве. Содержит инструкции, соглашения, руководства по использованию инструментов и определения рабочих процессов. Читается харнесом в начале каждой сессии. Переносимый между харнесами, поддерживающими это соглашение.

**MEMORY.md**
Файловое хранилище долгосрочной памяти. Содержит отобранные знания, которые нужны агенту между сессиями: предпочтения пользователя, детали проекта, извлечённые уроки. В отличие от проприетарных систем памяти, MEMORY.md человекочитаем, редактируем и переносим.

**ReAct (Reasoning + Acting)**
Паттерн промптинга, при котором модель чередует рассуждение («мне стоит поискать X») и действие (вызов инструмента). Создаёт цикл «Мысль → Действие → Наблюдение». Большинство современных харнесов для агентов реализуют вариант ReAct, часто неявно через цикл инструментов.

**Цепочка рассуждений (Chain-of-Thought, CoT)**
Техника промптинга, при которой модель стимулируется «думать пошагово» перед выдачей финального ответа. Расширенное мышление (как в режиме thinking Claude) — это реализация этого на уровне модели: модель генерирует внутренние токены рассуждений перед ответом.

**HaaS (Harness as a Service)**
Паттерн развёртывания агентных харнесов как управляемых облачных сервисов. API-шлюз направляет запросы к менеджерам сессий, которые диспетчизируют харнес-воркеры в изолированных средах. Обеспечивает мультитенантный хостинг агентов с изоляцией сессий и ограничениями ресурсов.

**Тонкий харнес (Thin Harness)**
Философия проектирования, при которой сам харнес минимален — только цикл инструментов, управление контекстом и оркестрация сессий — а предметная логика живёт в навыках. Харнес — универсальный движок; навыки обеспечивают экспертизу. Противоположность монолитных архитектур агентов.

**Толстые навыки (Thick Skills)**
Антоним «тонкого харнеса». Навыки содержат детальные инструкции, примеры и рабочие процессы для конкретных задач. Толстый навык для электронной почты может включать шаблоны, рекомендации по тону и специфические правила форматирования для платформ. Навык — это то, где живут предметные знания.

**Мультиагентность (Multi-Agent)**
Архитектура, в которой несколько ИИ-агентов совместно работают над задачей. Агенты могут иметь разные роли (исследователь, писатель, рецензент), разные модели или разные наборы инструментов. Требует оркестрации — определения, кто какой агент делает и как они общаются.

**Субагент (Sub-Agent)**
Агент, порождённый другим агентом для решения конкретной подзадачи. Родительский агент делегирует работу и получает результаты по завершении субагента. Субагенты имеют изолированные сессии, но могут разделять память или доступ к рабочему пространству. Ключевой паттерн для декомпозиции сложных задач.

**Песочница (Sandbox)**
Изолированная среда выполнения для безопасного запуска вызовов инструментов агента. Обычно это контейнер или виртуальная машина с ограниченной файловой системой, сетью и доступом к процессам. Предотвращает случайное (или умышленное) влияние агента на хост-систему. Необходима для инструментов выполнения кода.

**Сжатие контекста (Context Compression)**
Техники для умещения большего объёма информации в фиксированное контекстное окно. Включает суммаризацию старых сообщений, удаление избыточного контента, обрезку больших выводов инструментов и выборочную загрузку только релевантных файлов. Критически важно, когда история диалога превышает лимит контекста.

**Бюджет токенов (Token Budget)**
Стратегия распределения контекстного окна на разные цели: системный промпт, память, история диалога, результаты инструментов и вывод модели. Хорошо спроектированный бюджет токенов обеспечивает приоритет наиболее важной информации. Пример: 20% на системный промпт, 15% на память, 50% на историю, 15% на ответ.

**Сессия (Session)**
Одно непрерывное взаимодействие между пользователем и агентом. Сессия имеет начало (пользователь отправляет первое сообщение), серию обменов (обмен с вызовами инструментов) и конец (пользователь отключается или сессия истекает). Состояние сессии включает историю диалога, активные инструменты и любой накопленный контекст.

**Контрольная точка (Checkpoint)**
Снимок состояния агента в конкретный момент сессии. Включает историю диалога, состояния инструментов и промежуточные результаты. Позволяет возобновлять работу после сбоев или пауз без перезапуска с нуля. Критически важно для длительных задач, где прерывание на полпути обходится дорого.

**MCP (Model Context Protocol)**
Открытый протокол для подключения ИИ-моделей к внешним инструментам и источникам данных. Определяет стандартный интерфейс, чтобы инструменты (MCP-серверы) могли использоваться любым совместимым харнесом (MCP-клиент). Стремится решить проблему интеграции N×M — вместо интеграции каждого харнеса с каждым инструментом, обе стороны говорят на MCP.

**Оценка (Eval)**
Процесс измерения производительности агента по заданным критериям. Включает поведенческие тесты (вызвал ли агент правильные инструменты?), тесты результатов (был ли результат правильным?) и экономические тесты (сколько это стоило?). Оценки для агентов — это то, что модульные тесты для кода — не опциональны в production.

**Трассировка (Trace)**
Полная запись сессии агента: каждое отправленное сообщение, каждый вызванный инструмент, каждый полученный результат и финальный вывод. Трассировки используются для отладки (что пошло не так?), оценки (как справился агент?) и регрессионного тестирования (хуже ли новая версия?). Аналог лог-файла для агентов.
