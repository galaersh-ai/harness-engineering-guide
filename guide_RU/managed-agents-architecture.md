# Управляемые агенты: разделение мозга и рук

> **Ключевая идея:** Монолитный контейнер агента — где харнес, песочница и состояние сессии разделяют один процесс — это питомец, которого Вы не можете позволить себе потерять. Разделение мозга (харнес + LLM) от рук (песочница + инструменты) и сессии (журнал событий) превращает каждый компонент в заменяемый скот и открывает независимое масштабирование, восстановление после сбоев и изоляцию безопасности.

## Проблема монолита

Простейшая архитектура агента помещает всё в одну коробку:

```
┌──────────────────────────────────┐
│         Single Container          │
│                                   │
│  ┌─────────┐  ┌──────────────┐   │
│  │ Harness  │  │   Sandbox    │   │
│  │ (brain)  │  │ (code exec)  │   │
│  │          │  │              │   │
│  │ Session  │  │ Credentials  │   │
│  │ state    │  │ User files   │   │
│  └─────────┘  └──────────────┘   │
│                                   │
└──────────────────────────────────┘
```

Это работает для прототипов. Для продакшена это падает тремя предсказуемыми способами.

### Сбой 1: Проблема питомца

В аналогии [питомцы vs. скот](https://cloudscaling.com/blog/cloud-computing/the-history-of-pets-vs-cattle/), питомец — это именованный, вручную поддерживаемый сервер, которого Вы не можете позволить себе потерять. Скот взаимозаменяем — когда один умирает, Вы заменяете его.

Монолитный контейнер агента — это питомец. Если контейнер падает, сессия теряется. Если он становится неотвечающим, Вам приходится откармливать его обратно к жизни. Нельзя просто убить его и запустить новый, потому что состояние сессии — полная история того, что сделал агент — живёт внутри.

### Сбой 2: Слепое пятно отладки

Когда монолитный контейнер зависает, Ваше единственное окно — внешний поток событий. Но ошибка харнеса, потерянный пакет и упавший контейнер выглядят одинаково снаружи. Чтобы диагностировать реальную причину, инженеру нужен shell-доступ внутрь контейнера — но этот контейнер также хранит данные пользователя и учётные данные. Отладка становится инцидентом безопасности.

### Сбой 3: Граница безопасности

Недоверенный код, генерируемый агентом, работает в том же контейнере, что и учётные данные. Prompt-injection нужно только убедить агента прочитать собственные переменные окружения. Получив эти токены, атакующий может порождать неограниченные сессии. Узкая область действия помогает, но кодирует предположения о том, что модель не может сделать с ограниченными токенами — а модели постоянно становятся умнее.

## Трёхуровневая архитектура

Решение — декомпозировать монолит на три независимых слоя, каждый со своим жизненным циклом:

```
┌─────────────────────────────────────────────────────────┐
│                    Orchestration Layer                    │
│                                                          │
│  ┌──────────────┐  ┌──────────┐  ┌───────────────────┐  │
│  │    Brain      │  │ Session  │  │      Hands        │  │
│  │  (Harness +   │  │ (Event   │  │  (Sandbox A)      │  │
│  │   LLM calls)  │  │  Log)    │  │  (Sandbox B)      │  │
│  │              │  │          │  │  (MCP Tools)      │  │
│  │  Stateless   │  │ Durable  │  │  Disposable       │  │
│  └──────┬───────┘  └────┬─────┘  └────────┬──────────┘  │
│         │               │                  │             │
│         │  emitEvent()  │   execute()      │             │
│         ├──────────────►│◄─────────────────┤             │
│         │  getEvents()  │   provision()    │             │
│         ├──────────────►│                  │             │
│         └───────────────┴──────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

### Мозг (харнес)

Харнес выполняет агентный цикл: вызывает LLM, маршрутизирует вызовы инструментов, управляет контекстом. Критически важно, что харнес **бесостоятельный**. Он не хранит данные сессии в памяти. Всё персистентное попадает в журнал событий через `emitEvent()`.

Когда харнес падает, новый запускается, вызывает `wake(sessionId)`, извлекает журнал событий через `getEvents()` и возобновляется с последнего события. Нет потери данных. Нет откармливания. Харнес стал скотом.

### Сессия (журнал событий)

Сессия — это **только добавляемый журнал** всего, что произошло: вызовы LLM, результаты инструментов, сообщения пользователей, системные события. Он живёт вне харнеса и песочницы в персистентном хранилище.

Ключевые интерфейсы:
- `emitEvent(sessionId, event)` — запись события во время агентного цикла
- `getEvents(sessionId, options)` — чтение событий обратно (позиционная нарезка, фильтрация)
- `getSession(sessionId)` — получение метаданных и статуса

Сессия переживает и харнес, и песочницу. Если любой из них падает, сессия остаётся целой.

### Руки (песочница)

Песочницы — это среды выполнения, где агент запускает код, редактирует файлы и выполняет команды. Они создаются по запросу через `provision({resources})` и уничтожаются, когда больше не нужны.

Харнес вызывает песочницы так же, как любой инструмент: `execute(name, input) → string`. Если песочница падает, харнес ловит ошибку как неудачный вызов инструмента и передаёт её LLM. Модель может решить повторить попытку на свежей песочнице.

## Сессия ≠ контекстное окно

Это различие самое тонкое и самое важное в архитектуре.

**Сессия** — это полная, персистентная запись всего — потенциально миллионы токенов, охватывающих часы или дни работы агента. **Контекстное окно** — это подмножество этой записи, которое харнес выбирает для текущего вызова LLM — обычно 128K–200K токенов.

```
Session (append-only event log, durable)
┌─────────────────────────────────────────────────────────────┐
│ event_1 │ event_2 │ ... │ event_500 │ ... │ event_2000     │
└─────────────────────────────────────────────────────────────┘
                                    │
                          getEvents(slice)
                                    │
                                    ▼
                    Context Window (selected subset)
                    ┌───────────────────────────┐
                    │ system_prompt             │
                    │ event_1950 ... event_2000 │
                    │ (50 most recent events)   │
                    └───────────────────────────┘
```

Это разделение даёт три преимущества:

**1. Необратимые решения становятся обратимыми.** Сжатие (суммаризация старого контекста для освобождения места) необратимо уничтожает информацию. С персистентным журналом сессий харнес может перечитать любое прошлое событие — даже те, которые ранее были сжаты из контекстного окна. Если следующему ходу нужна информация из event_200, харнес может её извлечь.

**2. Инженерия контекста становится заботой харнеса.** Харнес решает, что попадает в контекстное окно: недавние события, суммаризация старых событий, системные промпты, релевантное содержимое файлов. Различные реализации харнеса могут использовать разные стратегии без изменения формата сессии. Сегодняшняя инженерия контекста может быть поуровневой обрезкой; завтрашняя может быть семантическим извлечением. Сессия остаётся той же.

**3. Мульти мозговые архитектуры становятся возможными.** Несколько харнесов могут одновременно читать один журнал сессий. Планирующий мозг может просматривать всю историю, пока исполнительный мозг фокусируется на недавних событиях. Оба читают из одного персистентного источника.

## Безопасность: учётные данные никогда не попадают в песочницу

Разделённая архитектура создаёт естественную границу безопасности. Учётные данные живут в хранилище (vault) вне песочницы. Агент никогда не обрабатывает токены напрямую.

Два паттерна:

**Привязка к ресурсу:** Для Git токен доступа используется во время инициализации песочницы для клонирования репозитория и настройки локального удалённого. `git push` и `git pull` работают изнутри песочницы, не видя токен.

**Хранилище + прокси:** Для внешних API (через MCP) OAuth-токены живут в безопасном хранилище. Агент вызывает MCP-инструменты через прокси. Прокси ищет учётные данные сессии в хранилище и делает внешний вызов. Песочница никогда не видит учётных данных.

```
┌───────────┐     ┌───────────┐     ┌───────────┐     ┌──────────┐
│  Agent    │────►│  MCP      │────►│  Vault    │────►│ External │
│ (sandbox) │     │  Proxy    │     │ (secrets) │     │ Service  │
│           │     │           │     │           │     │          │
│ No creds  │     │ Session   │     │ OAuth     │     │          │
│ here      │     │ → creds   │     │ tokens    │     │          │
└───────────┘     └───────────┘     └───────────┘     └──────────┘
```

Даже если prompt-injection убедит агента искать во всей среде, учётных данных для поиска нет.

## Производительность: улучшения TTFT

В монолитном дизайне каждая сессия — даже те, которые никогда не трогают песочницу — платили полную стоимость настройки контейнера: клонирование репозитория, запуск процесса, извлечение событий. Инференс не мог начаться, пока контейнер не был готов.

С разделённой архитектурой харнес запускается немедленно. Настройка песочницы происходит лениво через вызов инструмента, только если нужно. Результаты из production-развёртывания Anthropic:

| Метрика | Монолит | Разделённый | Улучшение |
|---------|---------|-------------|-----------|
| TTFT p50 | базовая | ~40% от базовой | **~60% снижение** |
| TTFT p95 | базовая | ~10% от базовой | **~90% снижение** |

Улучшение p95 впечатляющее, потому что худший сценарий запуска контейнера (холодные подтягивания, медленный диск) полностью удаляется из критического пути для сессий, которые не нуждаются в песочнице.

## Много мозгов, много рук

Архитектура естественно расширяется в обоих измерениях.

### Масштабирование мозгов

Масштабирование на множество параллельных сессий агента означает запуск множества бесостоятельных харнесов. Поскольку каждый харнес не хранит состояние (всё персистентное в журнале сессий), масштабирование горизонтальное и тривиальное. Нет общей памяти, нет распределённых блокировок, нет синхронизации состояния.

### Масштабирование рук

Один мозг может подключаться к нескольким песочницам. На практике это означает, что агент рассуждает о нескольких средах выполнения и решает, куда отправить работу:

```
┌──────────┐
│  Brain   │
│ (harness)│
└────┬─────┘
     │
     ├──► Sandbox A (Python environment)
     ├──► Sandbox B (Node.js environment)
     ├──► Sandbox C (Customer VPC)
     └──► MCP Tool Server (external APIs)
```

Ранние модели не могли с этим справиться — одновременное рассуждение о нескольких средах выполнения было слишком сложным. По мере улучшения возможностей моделей предположение об одноконтейнерности стало узким местом. Разделённая архитектура была готова к тому, чтобы модели догнали.

### Интеграция с VPC заказчика

Когда всё работало в одном контейнере, подключение к инфраструктуре заказчика означало сетевое соединение — VPC заказчика должен был быть доступен из нашего контейнера. С разделённой архитектурой песочница может работать внутри VPC заказчика, пока мозг работает в нашей инфраструктуре. Мозг общается с песочницей через стандартный интерфейс `execute()`, независимо от того, где физически работает песочница.

## Антипаттерны

### Хранение состояния сессии в харнесе

```python
# ❌ Session state in harness memory
class HarnessWithState:
    def __init__(self):
        self.events = []  # Lost if harness crashes
        self.current_plan = None
    
    def run_turn(self, message):
        result = self.llm.call(self.events + [message])
        self.events.append(result)  # Only in memory
```

Если харнес падает, всё теряется. Всегда отправляйте события в персистентный журнал сессий.

```python
# ✅ Session state externalized
class StatelessHarness:
    def run_turn(self, session_id, message):
        events = self.session_store.get_events(session_id)
        result = self.llm.call(events + [message])
        self.session_store.emit(session_id, result)  # Durable
```

### Учётные данные в песочнице

```python
# ❌ Passing API keys into the sandbox environment
sandbox.execute("export GITHUB_TOKEN=ghp_xxx && git push")
```

Один prompt-injection от утечки. Используйте вместо этого паттерн хранилище + прокси.

### Раннее provisionирование песочницы

```python
# ❌ Provision sandbox before knowing if it's needed
sandbox = provision_sandbox(resources=large_config)  # 30s startup
result = llm.call(prompt)  # Might not even need the sandbox
```

Provisionируйте лениво. Пусть LLM решает через вызов инструмента, нужна ли ему песочница и какая.

## Чеклист проектирования

Перед построением управляемой архитектуры агента:

1. **Бесостоятельность мозга** — может ли харнес упасть и перезапуститься без потери данных?
2. **Персистентность сессии** — хранится ли журнал событий вне мозга и рук?
3. **Заменяемость песочниц** — может ли любая песочница быть убита и заменена?
4. **Изоляция учётных данных** — недоступны ли секреты из песочницы?
5. **Ленивое provisionирование** — создаются ли песочницы по запросу, а не заранее?
6. **Стабильность интерфейсов** — достаточно ли общие интерфейсы мозг↔руки, чтобы пережить текущие реализации?

## Резюме

Монолитный контейнер агента — естественная начальная точка — всё в одной коробке просто. Но он создаёт питомцев, которых нельзя заменить, границы безопасности, которых нельзя обеспечить, и узкие места производительности, которых нельзя устранить.

Декомпозиция на три слоя — бесостоятельный мозг, персистентная сессия, заменяемые руки — решает все три. Мозг становится скотом (падает и перезапускается). Сессия становится единым источником правды (переживает всё остальное). Руки становятся ресурсами по запросу (provisionируются только когда нужны, уничтожаются когда завершены).

Это тот же паттерн, который операционные системы обнаружили десятилетия назад: виртуализируйте компоненты, делайте интерфейсы стабильными и позвольте реализациям свободно меняться под ними. Системный вызов `read()` не заботит, читает ли он с дисковой пачки 1970-х или с современного SSD. Интерфейс `execute()` не заботит, работает ли песочница локально, в VPC заказчика или на другом континенте.
