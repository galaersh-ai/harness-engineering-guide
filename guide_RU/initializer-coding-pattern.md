# Инициализатор + Coding-агент — двухфазный паттерн харнеса

Если Вы когда-либо давали модели-программисту нового поколения промпт вроде *«построй мне клон Claude.ai»* и уходили на весь день, Вы знаете, как это заканчивается: репозиторий с рабочей формой логина, наполовину готовым списком сообщений, `TODO: hook up streaming` посреди файла и очень уверенным коммит-сообщением. У модели закончился контекст.

Инстинкт — винить модель. Почти всегда виноват харнес.

Эта статья об одном конкретном паттерне, который решает этот класс ошибок: **разделите Вашего длительного агента на две фазы — инициализатор, который запускается один раз, и coding-агента, который запускается много раз, каждый раз работая с чистого листа над одной функцией.**

Он основан на [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) со ссылкой в [autonomous-coding quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding). Я хочу объяснить *почему* это работает, а не просто как это выглядит.

---

## Ключевая идея: сжатия недостаточно

Стандартный ответ на «контекстные окна конечны» — это сжатие. Суммаризуй старое, сохрани новое, повторяй вечно.

Сжатие нормально для двухчасового рефакторинга. Оно рассыпается для двухдневной постройки.

Направьте Opus 4.5 из Agent SDK на промпт высокого уровня вроде «построй клон Claude.ai» и наблюдайте:

- Сессия 1 пишет много кода и упирается в лимит контекста с полузаконченными функциями.
- Сессия 2 читает сжатую сводку — потерянную, повествовательную версию того, что произошло, — и восстанавливает намерение из неё.
- Сессия 3 читает сжатие сжатия. Информация деградирует геометрически.

К сессии 4 модель играет в «испорченный телефон» со своим прошлым «я». Она не знает, какие функции готовы, какие наполовину готовы, а какие она сгаллюцинировала в сводку.

**Сжатие — это внутриконтекстная память. Агенты длительных задач нуждаются в внемконтекстной памяти** — файлах, структурированном состоянии, git-истории. Вещах, которые агент перечитывает каждую сессию, а не тех, которые переносятся через суммаризацию.

Это тезис. Всё остальное — механика.

## Два режима сбоя, с которыми Вы столкнётесь

Прежде чем мы перейдём к паттерну, давайте определимся, что идёт не так без него. Два режима сбоя доминируют.

### (A) Тенденция «всё за раз»

Модель пытается сделать всё в одной сессии. Получив большой промпт, она пишет большое решение. Контекст заполняется. Она продолжает давить. Сессия заканчивается с наполовину реализованными функциями, `// TODO` разбросанными посреди функций, набором тестов, который не запускается, и без чистого коммита для отката. Модель не знает, что она сломалась — у неё закончилось место раньше, чем она могла проверить.

### (B) Преждевременная победа

Более изощрённо. Сессия 2 читает заметки о прогрессе от сессии 1: «реализованы логин, обмен сообщениями, стриминг». Модель смотрит на код, видит что-то, что *выглядит* как логин + обмен сообщениями + стриминг, и объявляет проект завершённым. Она не запускала его. Не тестировала поток. Она доверила нарративу.

Преждевременная победа — это двойник «всё за раз». «Всё за раз» падает, потому что модель пытается сделать слишком много; преждевременная победа падает, потому что следующая модель доверяет предыдущему оптимизму. Любой харнес для длительных задач должен решить **обе проблемы одновременно**. Вот почему наивное «просто цикль со сжатием» не проходит — сжатие подкрепляет нарратив предыдущей сессии, что категорически неправильно, когда этот нарратив чрезмерно оптимистичен.

## Двухфазный харнес

Решение структурное. Один агент не делает всё. У Вас две роли, с разными обязанностями и разными жизненными циклами.

```
 ┌──────────────────────────────────────────────────────────────┐
 │                        HARNESS DRIVER                        │
 │                     (your Python script)                     │
 └──────────────────────────────────────────────────────────────┘
                  │                              │
                  │ runs once                    │ runs N times
                  ▼                              ▼
 ┌──────────────────────────┐     ┌──────────────────────────────┐
 │    INITIALIZER AGENT     │     │        CODING AGENT          │
 │                          │     │                              │
 │ • read spec / brief      │     │ • pwd + read git log         │
 │ • write init.sh          │     │ • read claude-progress.txt   │
 │ • write feature_list.json│     │ • read feature_list.json     │
 │ • write progress file    │     │ • pick highest-priority unit │
 │ • git init + first commit│     │ • run init.sh (dev server)   │
 │                          │     │ • implement ONE feature      │
 │                          │     │ • e2e test via browser       │
 │                          │     │ • flip passes: true          │
 │                          │     │ • commit + update progress   │
 └──────────────────────────┘     └──────────────────────────────┘
              │                                  ▲
              │                                  │
              └───── clean state repo ───────────┘
                     (shared on disk)
```

Инициализатор запускается **один раз**. Его задача — превратить нечёткий бриф от человека в конкретный, машиночитаемый план плюс рабочий каркас.

Coding-агент запускается **много раз**, каждый вызов — новый процесс без памяти о предыдущем. Он получает память, читая файлы, которые предыдущие сессии записали на диск.

Два агента, две цели, одна общая файловая система.

---

## Агент-инициализатор

Инициализатор — единственная фаза, которая думает о всём проекте сразу. После её завершения никто больше не имеет глобального вида — каждая coding-сессия работает локально.

Четыре артефакта, ровно:

1. **`init.sh`** — один скрипт, настраивающий проект с холодного чекаута: установка зависимостей, запуск миграций, запуск dev-сервера в фоне, вывод URL. Каждая coding-сессия запускает это первой. Если `init.sh` сломан, остальное не важно.
2. **`feature_list.json`** — декомпозированный бэклог. Подробности формата в следующем разделе.
3. **`claude-progress.txt`** — короткие прозаичные заметки о том, где проект и что дальше. Обновляется в конце каждой сессии.
4. **Начальный git-коммит** — каркас, скрипты, список функций, файл прогресса — всё в `HEAD`. С этого момента каждая сессия заканчивается коммитом. `git log` — источник правды о том, что фактически выпущено.

Инициализатор не реализует функции. Он не выбирает цвета. Он накрывает стол.

## Проектирование списка функций: почему JSON лучше Markdown

«Это просто список задач, почему не чеклист в Markdown?»

Потому что когда Вы даёте Claude Markdown-файл, он обращается с ним как с прозой. Прозу можно редактировать. Модель с лучшими намерениями перепишет Ваши приоритеты, перефразирует описания, объединит две функции, потому что «так чище», и разделит одну на пять, потому что «проясняет намерение». Markdown-списки задач — болото для агентов.

JSON другой. Модели обучены обращаться с JSON как со структурированными данными — чем-то, откуда Вы читаете и пишете конкретные поля, а не переписываете наугад. В сочетании с явным правилом, что только одно поле доступно для записи, JSON становится почти неуязвимым внешним состоянием.

Разумная схема:

```json
{
  "project": "claude-ai-clone",
  "created_at": "2026-04-19T10:00:00Z",
  "features": [
    {
      "id": "auth-01",
      "title": "Email + password sign up",
      "description": "User can create account with email + password. Passwords hashed with bcrypt. Email uniqueness enforced.",
      "priority": 1,
      "depends_on": [],
      "acceptance": [
        "POST /api/signup returns 201 with session cookie",
        "duplicate email returns 409",
        "password stored as bcrypt hash, not plaintext"
      ],
      "passes": false
    },
    {
      "id": "auth-02",
      "title": "Email + password sign in",
      "description": "User can log in with credentials from auth-01.",
      "priority": 2,
      "depends_on": ["auth-01"],
      "acceptance": [
        "POST /api/signin with valid creds returns 200 + cookie",
        "wrong password returns 401",
        "signed-in user can load /app"
      ],
      "passes": false
    },
    {
      "id": "chat-01",
      "title": "Minimal chat page with message list",
      "description": "Authenticated user sees a chat page with past messages rendered from the DB.",
      "priority": 3,
      "depends_on": ["auth-02"],
      "acceptance": [
        "GET /chat renders <ul> of messages",
        "unauthenticated GET /chat redirects to /signin"
      ],
      "passes": false
    }
  ]
}
```

Дисциплина закодирована в системном промпте coding-агента:

> Вы можете изменять только поле `passes`. Недопустимо удалять или редактировать тесты, критерии приёмки, описания или структуру этого файла. Если Вы считаете, что функция определена неверно, остановитесь и сообщите — не переписывайте молча.

Эта формулировка не для проформы — она несёт нагрузку. «Недопустимо» читается моделью как жёсткое правило. На практике этого достаточно, чтобы файл оставался стабильным на протяжении десятков сессий.

`passes: false` — другая ключевая деталь. Это единственный рычаг агента. Когда сессия реализует функцию, проходит e2e-тесты и коммитит чисто, она переключает булево значение в `true`. Одно поле — не проза, не вайб — это то, как будущие сессии узнают, что фактически готово.

## Coding-агент: стартовый ритуал, а не произвол

Каждый раз, когда coding-агент просыпается, он запускает один и тот же пятиэтапный ритуал перед тем, как трогать код. Не опционально.

```
1. pwd                         → confirm we're in the project
2. git log --oneline -20       → what shipped, most recent first
   cat claude-progress.txt     → narrative notes from last session
3. cat feature_list.json       → what's done, what's next, what's blocked
4. bash init.sh                → install, migrate, start dev server
5. hit the app in a browser    → verify last session's work still works
```

Только после шага 5 агент выбирает функцию и начинает писать код.

Назовите это **контекстным бутстрапом** — внемконтекстным аналогом сжатия. Вместо наследования суммаризации агент реконструирует рабочий контекст из файлов на диске. Те же канонические источники, тот же порядок, каждая сессия.

Несколько моментов, которые стоит закрепить:

- **Git log перед заметками о прогрессе.** Git — ground truth; заметки о прогрессе оптимистичны. Если они расходятся, побеждает git.
- **`init.sh` не обсуждается.** Если dev-сервер не запускается, всё, что утверждала предыдущая сессия, не имеет значения.
- **E2e-проверка перед выбором новой функции.** Это то, что убивает преждевременную победу. Вы не доверяете «готово» предыдущей сессии — Вы открываете браузер.

На шаге 3 агент выбирает *одну функцию с наивысшим приоритетом, у которой `passes: false` и все зависимости выполнены*. Не две. Не «кластер». Одну. Затем останавливается, когда она зелёная и закоммичена.

Одна функция за сессию убивает оба режима сбоя одновременно: убивает «всё за раз», потому что контракт искусственно ограничивает область, и убивает преждевременную победу, потому что «готово» — не самодекларация, а e2e-тест, проходящий на чётко определённой единице.

## E2e-тестирование — слой доверия

Модульные тесты необходимы, но недостаточны. Их написал coding-агент — он не может быть единственным, кто их проверяет. Если модель неправильно поняла спецификацию, модульные тесты будут неправильными в той же форме, что и реализация, и пройдут так, что это ничего не значит.

E2e — это то, где Вы получаете независимый сигнал. Запустите реальное приложение, управляйте им как пользователь, смотрите, что происходит.

На практике это означает автоматизацию браузера. [Puppeteer MCP](https://github.com/modelcontextprotocol/servers) хорошо работает — агент может навигировать, заполнять формы, кликать и проверять DOM.

Проход проверки для `auth-01`:

```
→ navigate to http://localhost:3000/signup
→ fill #email with "test+{{timestamp}}@example.com"
→ fill #password with "correct-horse-battery-staple"
→ click button[type=submit]
→ assert url === "/app"
→ assert response cookie "session" is set
→ navigate to /signup again with same email
→ fill + submit
→ assert error text contains "already exists"
```

Три момента: (1) работает против реального сервера, запущенного `init.sh`, а не мока; (2) проверяет наблюдаемое поведение — URL, DOM, куки — а не внутренние функции; (3) достаточно дешёв для запуска каждую сессию, так что проверка бутстрапа «пережила ли работа прошлой сессии» — это ещё и e2e-проход уже зелёных функций.

Именно это делает `passes: true` значимым.

## Коммиты чистого состояния: git + файл прогресса вместе

В конце каждой coding-сессии две вещи происходят последовательно:

1. `git commit -m "feat(auth-01): email+password signup"` — код летит.
2. `claude-progress.txt` получает короткое дополнение — что выпущено, что сложно, на что следующей сессии стоит посмотреть первой.

Они делают разную работу. Git — слой **правды**: только добавляемый, с временнЫми метками, diff не обсуждается. `claude-progress.txt` — слой **подсказок**: заметки для следующего агента вроде «шаблоны email лежат в `/lib/email/`, я monkeypatch'нул SMTP-клиент для тестов, осторожно с гонкой в `/api/signin`, я обошёл через retry».

Оба читаются в начале каждой сессии. Ни одного недостаточно. Git без подсказок сух и труден для восстановления намерения. Подсказки без git плывут. Вместе они являются внемконтекстной памятью, заменяющей сжатие.

Оба должны завершать сессию в *чистом* состоянии — без незакоммиченных изменений, без наполовину написанных файлов, без сломанного dev-сервера. Если функция не готова к лимиту контекста, **сессия откатывается к последнему чистому коммиту и сообщает «не готово»**, а не коммитит сломанную работу. Так Вы держите git надёжным.

## Где этот паттерн подходит (а где нет)

Будьте честны относительно формы Вашего проекта, прежде чем прибегать к этому.

**Хорошо подходит:** веб-приложения (декомпозируемые по функциям, e2e-тестируемые), CLI (каждая подкоманда — естественная единица), внутренние инструменты / дашборды / админ-панели, пайплайны данных с дискретными этапами.

**Плохо подходит:** одиночные алгоритмические исследования вроде «спроектируй лучший механизм внимания» (не декомпозируется в `passes: true/false`), сильно связанные кодовые базы, где изменение одной функции вынуждает менять всё, всё, чья ценность живёт в *связях* между компонентами — корректность компилятора — в глобальных инвариантах, а не в «добавь лексер» + «добавь парсер».

Правило: если Вы можете записать критерии приёмки как «пользователь (или shell) делает X и наблюдает Y», этот паттерн работает. Иначе выберите другой харнес.

## Эскиз реализации

Сам харнес-драйвер мал. Основная работа — в двух системных промптах.

```python
# harness.py — pseudocode, omit the error handling for clarity
from pathlib import Path
import json, subprocess, time
from claude_sdk import run_agent

PROJECT = Path("./project")

def initialize(brief: str) -> None:
    """Run once. Turns a brief into a runnable scaffold."""
    run_agent(
        system_prompt=INITIALIZER_PROMPT,
        user_prompt=brief,
        workdir=PROJECT,
        allowed_tools=["write_file", "run_shell", "git"],
        # the initializer is expected to produce:
        #   init.sh, feature_list.json, claude-progress.txt, first git commit
    )
    assert (PROJECT / "feature_list.json").exists()
    assert (PROJECT / "init.sh").exists()

def next_feature() -> dict | None:
    features = json.loads((PROJECT / "feature_list.json").read_text())["features"]
    features.sort(key=lambda f: f["priority"])
    for f in features:
        if f["passes"]:
            continue
        if all(done(dep) for dep in f["depends_on"]):
            return f
    return None  # everything done

def done(feature_id: str) -> bool:
    features = json.loads((PROJECT / "feature_list.json").read_text())["features"]
    return any(f["id"] == feature_id and f["passes"] for f in features)

def code_one_feature() -> bool:
    feat = next_feature()
    if feat is None:
        return False
    run_agent(
        system_prompt=CODING_PROMPT,         # includes the startup ritual
        user_prompt=f"Implement feature {feat['id']}: {feat['title']}",
        workdir=PROJECT,
        allowed_tools=["read_file", "write_file", "run_shell", "git", "puppeteer"],
    )
    # agent is expected to leave a clean commit + updated progress file
    return True

if __name__ == "__main__":
    if not (PROJECT / "feature_list.json").exists():
        initialize(open("brief.md").read())
    while code_one_feature():
        time.sleep(1)  # fresh process each iteration — no state carried over
    print("all features passed.")
```

Два момента для внутреннего усвоения: **каждый вызов `run_agent` — это новый процесс** — нет общей памяти, нет общего контекстного окна, память — это файловая система. И **драйвер намеренно глуп** — он не выбирает функции, не валидирует работу, не интерпретирует результаты. Вся интеллектуальность — в промптах и артефактах.

## Заключительная мысль

Этот паттерн работает не потому, что инициализатор или coding-агент умны. Он работает потому, что **«что я знаю?» отвечается файлами на диске, а не содержимым контекстного окна.**

Каждый раз, когда Вы переносите состояние из внутриконтекстного во внемконтекстное, Вы получаете персистентность и ничего не теряете важного. Двухфазный харнес — это убеждённый способ сделать этот перенос, применённый к конкретной проблеме построения ПО за множество сессий.

Если Ваш агент длительных задач постоянно «забывает», решение почти никогда не в большем контекстном окне. Оно — в большем количестве файлов.
